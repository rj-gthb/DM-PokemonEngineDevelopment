# Pokemon Day II Engine System

This system supports Pokemon Day II battles by turning PokeAPI data into three usable engines:

- Team Engine: generates Gym Leader defending teams.
- Challenger Selection Engine: recommends challenger lineups against a Gym Leader team.
- Battle Prediction Engine: predicts a winner before battle and records the actual result after battle.

## How To Run

Start the Flask backend:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r server\requirements.txt
python server\app.py
```

Start the React frontend in another terminal:

```powershell
npm install
npm run dev
```

Open the Vite URL, usually:

```text
http://localhost:5173
```

## Data Source

All Pokemon data is retrieved from PokeAPI. The frontend fetches base Pokemon data and fetches detailed data when teams are generated or searched.

The current assigned section pool is:

| Section | Allowed Native Regions |
| --- | --- |
| 3ISA | Hoenn, Sinnoh, Galar |

Native region is mapped by original National Dex ID range:

| Region | Generation | ID Range |
| --- | --- | --- |
| Hoenn | Gen 3 | 252-386 |
| Sinnoh | Gen 4 | 387-493 |
| Galar | Gen 8 | 810-898 |

## Battle Restrictions

The engines always exclude restricted Pokemon and forms:

- Legendary Pokemon
- Mythical Pokemon
- Ultra Beasts
- Mega forms
- Primal forms
- Gigantamax forms

Restricted battle mechanics such as Mega Evolution, Dynamax, Gigantamax, Z-Moves, and Terastallization are not generated by the system.

## Engine Logic

### Team Engine

Inputs:

- Native region
- Type specialization
- PokeAPI Pokemon data
- Restriction filter

Output:

- Six eligible Gym Leader Pokemon when enough native type-specialist Pokemon exist
- Native region, type, role, and stats shown per Pokemon

Current logic:

1. Load Pokemon from the assigned native region.
2. Remove restricted Pokemon.
3. Keep Pokemon matching the selected Gym type specialization.
4. Score candidates using Base Stat Total, secondary type utility, and role diversity.
5. Select up to six Pokemon.

### Challenger Selection Engine

Inputs:

- Gym Leader team
- Challenger native region
- Model strategy
- PokeAPI Pokemon data

Available model strategies:

- K-Means: groups candidates into role/stat clusters for team variety.
- K-NN: finds Pokemon close to useful counter archetypes.
- Cosine Similarity: compares stat profiles against an ideal counter shape.
- Gower Distance: compares mixed features such as stats, type, and region.
- Decision Tree: classifies candidates into battle roles.
- Random Forest: ranks candidates through multiple weighted scoring trees.
- Rule-Based Scoring: uses type advantage, resistance, weakness, and stats.

### Battle Prediction Engine

Inputs recorded before battle:

- Match ID
- Gym Leader name
- Challenger name
- Gym region and type
- Challenger region
- Gym team
- Challenger team
- Models used

Prediction output:

- Predicted winner
- Confidence score
- Prediction reason
- Timestamp

Ground truth recorded after battle:

- Actual winner
- Correct / incorrect result
- Replay link
- Screenshot or photo link
- Final score
- Number of turns
- Timestamp

## Metrics

The analytics dashboard tracks:

- Accuracy
- Precision
- Recall
- F1-score
- Confusion matrix
- Brier score
- Log loss
- Model usage and challenger model success

Positive class is defined as `challenger wins`.

## Logs and Audit Trail

The current implementation stores prediction logs, pending battles, completed battles, ground truth, and metrics in browser `localStorage`.

For final deployment, this can be upgraded to SQLite, CSV, Google Sheets, or another persistent storage option.

## Limitations

- Current models are heuristic, not trained from a historical battle dataset yet.
- Logs are stored locally in the browser unless exported.
- PokeAPI requests require internet access unless a local cache is added.
- Full battle mechanics, movesets, abilities, EV spreads, and held items are simplified.

## Suggested Improvements

- Add SQLite or CSV persistence for logs and audit records.
- Add a cached PokeAPI dataset for event reliability.
- Train real models once enough battle outcomes are collected.
- Add charts for model success rate and confidence calibration.
- Add a formal data pipeline diagram to the final documentation.
